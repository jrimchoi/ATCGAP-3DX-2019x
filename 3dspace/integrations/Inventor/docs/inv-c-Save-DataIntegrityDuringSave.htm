Data Integrity During Save

While saving Inventor data, if any data is not supported by ENOVIA then an error message is displayed and the error logs are saved as a text file, Error.txt in the <InventorIntegration Install Folder>.

The user must confirm whether to continue saving ignoring this data or to resolve the data integrity before data is saved to ENOVIA. If ignored, the user can save all other data except the parent node.

If the user cancels, then the save process is aborted and the user can open the text file, resolve the data, and start the save process again.

It is recommended to cancel save, resolve the data structure, and then start checkin again.

Issues for certain data types and the method to resolve the data integrity issues are listed in the following table:

Data Type

Method to Resolve

Duplicate iPart files: Same iPart files exist in the root folder i.e., the folder where Part Factory exists in sub-folder or the folder with the Part Factory name as well.

All iParts in a workspace should be either in root folder or in sub-folder

Missing iPartFactory files: Some iParts do not have associated iPartFactory files in the workspace. This creates a problem in creating a complete and correct family table data for ENOVIA.

Identify the missing iPart Factory files and place them in the root folder where the iParts in the Sub-folder exists.

Renamed iParts: Some iParts are renamed out side the iPartFactory. Such iParts do not have an entry in their corresponding iPartFactory tables and thus its family table data cannot be retrieved.

Open the corresponding Part Factory , identify the iPart that is referred from the iPart Factory and add the renamed name in the list of iParts of the factory.

File Name column not set in iPartFactory table by default in Inventor 9: Some iParts have their names based on a particular column [say Part Number column] in the table while the same column is not set as "File Name Column". This creates a problem in creating a complete and correct family table data for ENOVIA. The iParts existing in DB are treated as New because the file names generated are different.

Update the filenames in the column to the already existing ENOVIA names or rename ENOVIA names to the names mentioned in the list of iParts.

Custom iPart files: Custom iPart files generated in a location other than the active workspace where iPart Factory file exists are shown as unresolved in Inventor after Save.

The custom iParts must be generated in the active workspace where the iPartFactory file exists. Custom iParts files generated at any other location is not supported.

iPart Factory is directly connected to other types.

The specific iPart of the iPart Factory should be connected to the other types.
