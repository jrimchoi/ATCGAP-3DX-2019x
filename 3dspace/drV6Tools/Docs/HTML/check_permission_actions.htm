Check Permission Actions


Contents

The 3dxtools Check Permission Actions provides the ability to test the assignment on context user or state access of the current object. If the context user doesn’t have the defined assignment or access on current object then the action will not pass and further child actions will not be executed.

Check permission action will not block the current operation. You can define the failAction criteria to continue executing parallel actions or abort the operation

 

Configuration and Settings

To setup a 3dxtools Check Permission Actions:

1. Setting Configuration ➣

•Add high level business logic in below format to any Action Activity object Settings attribute.

<setting>

 <actions>

         <checkPermission>  

                 <name>CheckUserHasPermission</name>

                 <failAction>FailAndContinue</failAction>

                 <failMessage>The logged in user ${usera:fullname} is not a member of the 'Product Management' group and                      'DRLTools Admin' role..</failMessage>

                 <roles>

                         <role>DRLTools Admin</role>

                 </roles>

                 <groups>

                         <group>Product Management</group>

                 </groups>

                 <actions/>

         </checkPermission>

 </actions>

</setting>

 

2. Setting Details  ➣

Action

Description

name

Any meaningful value. It is recommended to start with “If”.

failAction

Define the execution criteria if object doesn’t pass the configured check. It can be Passed, NotPassed, FailAndStop and FailAndContinue.

FailAndStop will block the operation and stop executing parallel actions.

FailAndContinue will block the operation and allow executing parallel actions.

Only FailAndStop and FailAndContinue actions will display error message to the user.

failMessage

User friendly error message to be displayed on action failure. You can use macros for basics and attribute values. Macros will be evaluated on context object id

roles

Roles to be checked. System will check isassigned[RoleName] on context user.

groups

Groups to be checked. System will check isassigned[GroupName] on context user

associations

Associations to be checked. System will check isassigned[AssociationName] on context user.

accesses

Accesses to be checked. System will check current.access[AccessName] on context object

 

Within assignment or access operation will be AND operator. If any of the assignment\access is FALSE then action will return false.

Inter assignment or access operation will be AND operator. i.e. if Roles, Groups and Accesses defined then all three conditions must be satisfied

actions

child actions configuration

 

•For child actions, add any Action Activity you wish to run from this event.
